# Git 브랜치 (Branch)

## 1. 깃 브랜치란?

브랜치는 **독립된 개발 환경(작업 흐름)** 을 만들어주는 기능이다. 하나의 원본(코드베이스)에서 갈라져 나와, 서로 영향을 주지 않고 병렬로 작업할 수 있게 해준다.

## 2. 브랜치의 장점

- **원본에 대해 안전하다.** 독립된 개발 환경을 형성하기 때문에, 브랜치에서 무슨 작업을 하든 원본(마스터)에는 영향을 주지 않는다.
- **체계적인 협업과 개발이 가능하다.** "하나의 작업 = 하나의 브랜치" 원칙으로 나눠 진행하면, 작업 단위가 명확해지고 여러 사람이 동시에 협업하기 쉬워진다.

## 3. 브랜치를 왜 써야 하나?

> **상황: 상용 중인 서비스에 에러가 발생했다. 어떻게 해결할까?**

- 브랜치를 통해 **별도의 작업 공간**을 만들어 에러를 수정한다.
- 이 작업은 **마스터에 영향을 미치지 못한다.** (서비스는 그대로 돌아감)
- 에러가 해결되면 그때 **마스터에 반영(병합)** 한다.

즉, 위험한 작업을 안전하게 격리해서 진행하고, 검증된 결과만 반영할 수 있다는 것이 브랜치를 쓰는 핵심 이유다.

## 4. 마스터(master) 브랜치의 역할

- **디폴트(default) 브랜치**다. (저장소를 만들면 기본으로 생김)
- 모든 작업의 **기준점**이 된다.
- 다른 브랜치의 변경사항을 **코드 리뷰를 통해 통합(병합)** 하는 대상이다.

---

## 5. 브랜치 명령어

| 명령어 | 설명 |
| --- | --- |
| `git branch` | 로컬 브랜치 조회 |
| `git branch -r` | 원격(remote) 브랜치 조회 |
| `git branch -a` | 로컬 + 원격 전체 조회 |
| `git branch <이름>` | 새 브랜치 생성 |
| `git branch -d <이름>` | 브랜치 삭제 (병합된 브랜치만, 안전한 삭제) |
| `git branch -D <이름>` | 브랜치 강제 삭제 (병합 안 됐어도 삭제) |

> `-d`는 안전 삭제, `-D`는 강제 삭제. 대문자일수록 강력(위험)하다고 기억하면 편하다.

## 6. 깃 스위치(switch) 명령어

브랜치를 이동(전환)할 때 쓰는 명령어다.

| 명령어 | 설명 |
| --- | --- |
| `git switch <브랜치명>` | 해당 브랜치로 전환 |
| `git switch -c <브랜치명>` | 새 브랜치 **생성 후 전환** (create) |
| `git switch -c <브랜치명> <커밋아이디>` | 특정 커밋에서 새 브랜치 생성 후 전환 |

### 궁금증: `checkout` vs `switch` 차이

- 원래는 `git checkout`이 **브랜치 전환 + 파일 복원**을 모두 담당했다. 기능이 너무 많아 헷갈렸다.
- 그래서 역할을 둘로 쪼갰다:
  - **`git switch`** → 브랜치 전환 전용
  - **`git restore`** → 파일 복원 전용
- 즉 `switch`는 checkout의 "브랜치 이동" 기능만 떼어낸, 더 명확한 명령어다.

## 7. HEAD에 대해서

- **HEAD**는 "지금 내가 어느 브랜치(커밋)에 있는지"를 가리키는 **포인터**다.
- 브랜치를 이동한다는 것은 결국 **HEAD가 다른 브랜치를 가리키도록 옮기는 것**이다.
- 보통 HEAD → 현재 브랜치 → 최신 커밋 순으로 가리킨다.

## 8. 깃 스위치 주의사항

- 스위치하기 전에 **워킹 디렉토리의 파일이 모두 최소한 트래킹/커밋되어 있는지** 확인해야 한다.
- **스테이징되지 않은(또는 커밋 안 된) 변경사항을 들고 스위치하면?** 상황에 따라 결과가 갈린다:
  - **① 충돌 없는 경우** → 변경사항이 그대로 따라와서 새 브랜치에서도 그 수정 내용이 보인다.
  - **② 충돌하는 경우** → 대상 브랜치의 파일과 겹쳐 Git이 스위치를 **거부(에러)** 한다. → 먼저 커밋하거나 `git stash`로 잠시 치워둬야 한다.
  - **③ 실수 방지 팁** → 이동 전에 `git status`로 워킹 디렉토리를 깨끗하게(clean) 만들어두는 습관.

## 9. 깃 머지 (Merge)

브랜치의 작업 내용을 합치는 것. 크게 두 종류가 있다.

### 패스트 포워드 머지 (Fast-forward Merge)

- **"빨리감기" 머지**다.
- 갈라진 이후 **한쪽 브랜치에만 커밋이 쌓이고**, 다른 쪽은 그대로일 때 발생한다.
- 새 커밋을 만들지 않고, **HEAD(수신 대상)를 병합 대상 커밋 위치로 그냥 옮겨(감아)** 준다.
- 헤드가 앞으로만 달려나가는(직선으로 이어지는) 형태.

### 3-웨이 머지 (3-way Merge)

- 양쪽 브랜치가 **각자 따로 커밋을 쌓았을 때** 사용한다.
- **3개의 커밋**을 사용한다: 병합할 브랜치 커밋 2개 + 공통 조상 커밋 1개.
- 병합 결과로 **새로운 커밋(merge commit) 하나가 생긴다.**
- 머지 시 **Vim 에디터**가 뜨면서 기본 커밋 메시지를 그대로 쓸지/수정할지 물어본다.

---

## 10. Vim 에디터 사용법 (머지 커밋 메시지 작성용)

> Vim은 **마우스가 안 먹는다.** 키보드로만 조작하므로 기본은 알아야 한다.

### 두 가지 모드

1. **입력 모드 (Insert Mode)** — 텍스트를 입력하는 상태. `i`를 눌러 진입.
2. **명령 모드 (Command Mode)** — 저장/종료 등 명령을 내리는 상태. `:`(콜론)을 입력해서 명령을 친다.

### 자주 쓰는 흐름

| 목적 | 키 입력 |
| --- | --- |
| 입력 모드 진입 | `i` |
| 입력 모드 빠져나오기 | `Esc` |
| 저장하고 종료 | `:wq` (write + quit) |
| 저장 안 하고 강제 종료 | `:q!` |

> **가장 간단한 방법:** 기본 메시지를 그냥 쓸 거라면 → `Esc` 누르고 → `:wq` 입력 → `Enter`. 끝.

## 11. 머지 컨플릭트 (Merge Conflict)

양쪽 브랜치가 **같은 부분을 다르게 수정**했을 때 Git이 자동으로 합치지 못하고 충돌이 난다.

### 충돌 표시 방식 (시각적 마커)

```text
<<<<<<< HEAD
현재 브랜치의 내용
=======
병합 대상 브랜치의 내용
>>>>>>> 브랜치명
```

- `<<<<<<<` ~ `=======` : 현재(HEAD) 쪽 내용
- `=======` ~ `>>>>>>>` : 병합해 오는 쪽 내용

### 해결 순서

1. 충돌 난 파일을 열어 마커(`<<<`, `===`, `>>>`)를 보고 **원하는 형태로 직접 정리**한다. (마커도 모두 지운다)
2. 정리 끝나면 `git add`로 스테이징.
3. `git commit`으로 마무리한다. (충돌 해결 커밋)
   - 이때 다시 **Vim이 뜨면** 위의 Vim 사용법대로 저장·종료하면 된다.

## 한눈에 보는 핵심 요약

- 브랜치 = **안전한 독립 작업 공간**, 협업의 기본 단위
- 마스터 = **기준점 & 디폴트**, 검증된 것만 병합
- 이동 = `git switch` (HEAD를 옮기는 것) / 생성+이동 = `-c`
- 머지 = **패스트포워드**(직선, 새 커밋 X) vs **3-웨이**(갈라짐, 새 커밋 O)
- 충돌 나면 → 마커 정리 → `add` → `commit`
- Vim 탈출 = `Esc` → `:wq`
