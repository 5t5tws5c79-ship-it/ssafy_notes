# 유닛 테스트 (Unit Test)

## 1. 왜 필요한가?

AI는 코드를 완벽하게 작성하지 않는다. 새 기능을 추가하다 보면 **기존에 잘 돌아가던 기능이 망가지는 경우**가 생긴다.

이렇게 잘 되던 게 다시 깨지는 걸 **회귀 버그(Regression Bug)** 라고 한다. 회귀 버그가 많아질수록 TDD(테스트 주도 개발)의 필요성이 커진다.

## 2. 검증 방법 3가지

| 방법 | 문제점 |
|------|--------|
| 손으로 모두 눌러본다 | 번거롭다. 매번 반복해야 함 |
| 다 만들고 눌러본다 | **언제부터** 버그가 생겼는지 모름 |
| 매번 테스트로 잠궈두고, 개발할 때마다 테스트 코드를 돌린다 | ✅ 정답 |

> 핵심은 **"잠궈둔다"** 는 것. 한 번 통과시킨 기능은 테스트로 고정해두고, 이후 개발할 때마다 테스트를 돌려 깨졌는지 즉시 확인한다.

## 3. 유닛 테스트란?

**컴퓨터에게 기능 단위로 검증시키는 방법.**

사람이 일일이 눌러보는 대신, 기능 하나하나를 코드로 검증한다.

### AAA 패턴

테스트 코드는 보통 세 단계로 구성된다.

| 단계 | 이름 | 하는 일 |
|------|------|---------|
| 준비 | **A**rrange | 테스트에 필요한 데이터·객체 세팅 |
| 실행 | **A**ct | 검증할 기능을 실제로 호출 |
| 검증 | **A**ssert | 결과가 기대값과 같은지 확인 |

```js
test('장바구니 합계를 계산한다', () => {
  // Arrange - 준비
  const cart = new Cart();
  cart.add({ price: 1000, qty: 2 });

  // Act - 실행
  const total = cart.getTotal();

  // Assert - 검증
  expect(total).toBe(2000);
});
```

## 4. 단점과 그 해결

- **개발 시간이 증가한다** → 생산성이 떨어진다
- 하지만 **AI가 테스트 코드를 대신 작성**해주면서, 개발 시간 증가 문제가 상당 부분 해결됨

## 5. 주의사항

- ⚠️ **테스트 코드를 통과했다 = 버그가 없다**가 아니다
  - 테스트는 내가 작성한 케이스만 검증한다. 생각하지 못한 케이스는 여전히 뚫려 있다.
- ⚠️ **테스트 작성에 너무 많은 시간을 쏟지 말 것**
  - 오류가 발생하면 안 되는 **핵심 로직만** 테스트 코드로 잠근다.
- ⚠️ **테스트 커버리지가 높다고 좋은 게 아니다**
  - 커버리지는 "실행된 코드 비율"일 뿐, "제대로 검증했는지"를 뜻하지 않는다.
  - 숫자 채우기용 테스트는 유지보수 부담만 늘린다.

## 한 줄 요약

> 핵심 로직을 테스트로 잠궈두고, 개발할 때마다 돌려서 회귀 버그를 즉시 잡는다. 단, 커버리지 숫자에 집착하지 말 것.
