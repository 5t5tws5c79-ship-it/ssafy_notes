# Git 워크플로

---

## 1. 피처 브랜치 워크플로 (Feature Branch Workflow)

### 개념 — 공유 저장소 모델 (Shared Repository Model)

- 각 사용자가 **원격 저장소의 소유권(쓰기 권한)을 공유받는** 방식
- 중앙 원본 저장소는 **하나**
- 기능 개발은 master가 아닌 **별도 브랜치**에서 진행
- 각자 브랜치를 만들어 **중앙 저장소로 직접 푸시**

```bash
git push origin feature/login
```

### 핵심 규칙

> **master는 오직 머지를 통해서만 올라간다.**
> 직접 푸시 ❌ → PR을 올리고, 리뷰를 거쳐 머지 ⭕

### 전체 흐름

1. **콜라보레이터 추가** — 저장소 소유자가 팀원에게 권한 부여
2. **클론** — 중앙 저장소를 로컬로 복사
3. **브랜치 생성** — master에서 분기
4. **작업 후 푸시** — `git push origin feature/login`
5. **PR 생성**
6. **코드 리뷰** — 검토 & 피드백
7. **PR 통과 → 머지**
8. **master 다시 pull** — 머지된 최신 상태 반영
9. **로컬 브랜치 삭제** — 원격에서 병합 완료된 브랜치 정리

```bash
git checkout master
git pull origin master
git branch -d feature/login
```

## 2. PR (Pull Request) 란?

- **원본에 합병해달라고 요청하는 기능**
- 협업을 위한 **핵심 소통 도구**

### 왜 중요한가

- **코드 리뷰의 장을 마련한다**
  - 다른 사람이 내 코드를 검토할 수 있음
  - 피드백을 주고받을 수 있음
- 바로 머지되는 게 아니라 **한 단계 거름망** 역할

### 트러블슈팅 — PR 알림이 안 뜰 때

푸시하면 보통 GitHub가 자동으로 "Compare & pull request" 배너를 띄워준다.
안 뜬다면 수동으로 만들면 됨:

1. **Branches 탭** 진입 → 브랜치 개수가 달라진 걸 확인
2. 해당 **브랜치로 들어가기**
3. **`...` (점점점) 버튼** 클릭
4. **New Pull Request** 선택

## 3. 포킹 워크플로 (Forking Workflow)

### 포크(Fork) 란?

- **다른 사람의 원격 저장소를 자신의 계정으로 복사하는 것**
- 복사본은 **내 소유** → 권한 없이도 마음껏 푸시 가능

### 전체 흐름

1. 원본 저장소를 **내 계정으로 포크**
2. **포크한 저장소**를 클론
3. 브랜치 생성 후 작업
4. **포크 뜬 곳(내 저장소)으로 푸시**
5. 거기서 **원본 저장소로 PR 전송**
6. 원본 관리자가 리뷰 후 머지

> 핵심: **포크 → 내 저장소에 푸시 → 원본으로 PR**

---

## 4. 두 워크플로 비교

| 구분 | 피처 브랜치 | 포킹 |
|---|---|---|
| **권한** | 원격 쓰기 권한 필요 (콜라보레이터 등록) | 권한 불필요 |
| **저장소 수** | 1개 (중앙 원본) | 2개 (원본 + 내 포크) |
| **푸시 대상** | 중앙 원본 저장소 | 내 계정의 포크 저장소 |
| **PR 방향** | 브랜치 → master (같은 저장소 내) | 내 포크 → 원본 저장소 |
| **적합한 경우** | 팀 내부 협업 | 오픈소스 / 외부 기여자 |

## 한 줄 요약

- **피처 브랜치**: 같은 집 안에서 방(브랜치) 나눠 쓰기
- **포킹**: 남의 집을 통째로 복사해와서 작업 후, 원래 주인에게 "이거 반영해주세요" 요청하기
- **공통점**: 둘 다 최종 반영은 **PR → 리뷰 → 머지**
